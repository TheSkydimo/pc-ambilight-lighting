# PC Ambilight Lighting 网站结构与内容重设计（V2）

## 1. 重设计目标

本版以“先决策、后细节”为核心，重构站点信息架构与页面内容顺序，解决旧版存在的重复、层级混乱和转化路径分散问题。

目标：
- 用户 10 秒内理解：这是什么、该选什么、下一步做什么。
- 用户 3 次点击内完成：认知 -> 选型 -> 下载/安装。
- 形成清晰内容闭环：`Best Picks -> Comparisons -> Guides -> Download`。
- 保持内容层、数据层、UI 层、逻辑层解耦，便于长期迭代。

---

## 2. 用户路径（先定义结构，再填内容）

四类核心用户路径：
- 认知型：不了解 Ambilight，需要原理与价值判断。
- 选型型：需要快速比较不同软件方案。
- 操作型：已决定方案，想马上安装配置。
- 排错型：已使用中，遇到延迟、颜色偏差、识别失败等问题。

路径映射：
- 认知型 -> `Home` / `Guides: 入门`
- 选型型 -> `Best Picks` / `Comparisons`
- 操作型 -> `Download` / `Guides: 安装`
- 排错型 -> `Guides: 排错` / `FAQ`

## 4. 首页结构（只做分发，不做信息堆叠）

### 4.1 首页模块顺序

1) Hero（价值主张 + 双 CTA）
- H1：`PC Ambilight Lighting for Gaming and Daily Setup`
- 副标题：强调“低延迟、稳定、易上手”。
- CTA1：`查看推荐方案`
- CTA2：`开始安装`

2) Top Picks（先给答案）
- 固定 3 卡片：Skydimo、OpenRGB、SignalRGB。
- 每卡只保留：一句话结论、适合人群、进入详情。

3) 决策向导（按人群分流）
- 新手 -> 入门安装
- 游戏用户 -> 低延迟优化
- 多品牌硬件用户 -> 兼容矩阵

4) 热门对比入口
- `Skydimo vs OpenRGB`
- `Skydimo vs SignalRGB`

5) 常见问题快答（3-5 条）
- 每条仅 2-3 句，提供“查看完整排错步骤”链接。

6) 信任信息
- 更新日期、测试环境、披露说明。

### 4.2 首页约束

- 每个模块 1 个核心动作，避免并列 CTA 过多。
- 首页不放大段技术细节，细节全部下沉到内页。

## 5. 核心页面内容框架

## 5.1 Best Picks（主决策页）

页面目标：用户 3 分钟内完成“选哪个”。

固定结构：
- TL;DR 结论（首屏）
- 推荐列表（Skydimo 固定首位）
- 场景推荐（新手 / 游戏 / 多品牌）
- 不适合人群与替代方案
- 下一步操作（下载或安装）

卡片字段标准：
- 产品名
- 核心优势（1 行）
- 适合人群
- 注意事项（1 行）
- CTA：查看评测 / 进入下载

## 5.2 Comparisons（对比决策页）

页面目标：降低“选错后重装”的成本。

固定结构：
- 先结论（谁更适合什么场景）
- 指标表（易用性 / 稳定性 / 兼容性 / 资源占用 / 学习成本）
- 迁移成本说明（从 A 切到 B）
- 最终建议（按用户类型）

写作规则：
- 先场景结论，再解释指标。
- 不做无意义参数堆叠。

## 5.3 Guides（安装与排错中心）

页面目标：承接操作型和问题解决型流量。

目录结构：
- 入门与原理
- 安装与初始化
- 效果调优（亮度、同步、延迟）
- 故障排查（识别失败、闪烁、颜色偏差、卡顿）

单篇模板：
- 目标
- 前置条件
- 步骤
- 验证
- 常见错误与修复
- 下一步推荐阅读

## 5.4 Download（转化落地页）

页面目标：最短路径完成下载选择。

页面结构：
- 版本选择（Skydimo 首位）
- 平台入口（官网 / Microsoft Store / Steam）
- 版本差异说明（谁该选哪个）
- 安装后第一步（跳转到对应指南）

按钮文案统一：
- `下载 Skydimo 官网版`
- `在 Microsoft Store 获取`
- `在 Steam 获取`

## 5.5 FAQ（快答页）

页面目标：快速止损，导流到完整教程。

问题池建议：
- Ambilight 会影响游戏帧率吗？
- 为什么设备无法识别？
- 为什么会有延迟或颜色不准？
- 如何选择灯带与控制器？

回答规则：
- 每条先给短答案，再给“详细步骤”链接。

## 5.6 Reviews（深度评测页）

页面目标：为高意向用户提供可信证据。

单篇结构：
- 结论先行（适合 / 不适合）
- 功能与边界
- 性能与稳定性
- 兼容性与已知问题
- 同类对比与替代建议

## 6. 内容与工程分层（高内聚、低耦合）

分层职责：
- Content 层：Markdown/MDX（文章正文）
- Data 层：JSON（评分维度、兼容矩阵、FAQ）
- UI 层：可复用组件（卡片、对比表、FAQ 折叠）
- Logic 层：筛选、排序、聚合、评分映射（纯函数）

约束：
- UI 不直接读取原始数据结构，只消费 ViewModel。
- 模板字段统一，避免页面各自定义字段。
- 新增页面优先复用模板，不复制旧页面实现。
